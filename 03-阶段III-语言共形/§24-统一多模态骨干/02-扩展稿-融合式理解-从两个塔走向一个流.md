# §22

标题: 融合式理解
类型: 主章
要点: 融合式编码器让图文在编码器层面相遇；ITC + ITM + LM 多损失把对齐、匹配、生成压到同一骨干——但骨干仍是专用 VL Transformer，吃不下 LLM 缩放红利

# 阶段 III · 融合式理解：从两个塔走向一个流（ViLT / ALBEF / BLIP / CoCa / BEiT-3）

作者 / 机构: Kim 等 (NAVER) / Li 等 (Salesforce) / Li 等 (Salesforce) / Yu 等 (Google) / Wang 等 (Microsoft)
共形维度: 载体, 度量, 材料
关键贡献: 用融合式编码器（cross-attention 或 concat + self-attention）让图文两个模态在编码器层面直接相遇；用 ITC + ITM + LM 多损失组合把对齐、匹配、生成同时压到一个骨干里。从「两个塔」走向「一个流」的第一阶段完成。
年份: 2021–2023
序号: 22
残差 / 它催生了什么: 编码器层面的融合达成，但这层融合发生在「VL 专用 Transformer」里——没用上预训练大语言模型的推理能力；模型规模被专用架构卡在几百 M 参数级，吃不下 LLM 的缩放红利；生成仍是 caption 级，不是指令级 → 催生 §23 冻结 LLM + 视觉适配（Flamingo / BLIP-2 / Q-Former）让视觉去对接已有的大语言模型；生成指令化留给 §24。
类型: 架构 / 预训练
阶段: III · 语言共形

🎯
**主题**：§21 让两个塔在损失处相遇。这一节要回答的是：如果让两个塔在**编码器**里相遇，会发生什么？需要什么新机制？代价是什么？五篇一起回答这三个问题。关键的对照线始终是 §21：它们在**结构**上打开了 CLIP 两塔的锁，但在**规模**上又碰上了新的锁——这条新锁要等到 §23 才被打开。

## 1 · ViLT —— *ViLT: Vision-and-Language Transformer Without Convolution or Region Supervision* (Kim 等, NAVER, ICML 2021)

**共形贡献**：把视觉端的卷积 backbone 和区域监督全部拆掉，直接用 ViT 式的 patch embedding，然后把图像 patch token 与文本 token **拼成一个序列**投入同一个 Transformer。一下子把「图文融合」最单纯的版本写出来了：**序列拼接 + self-attention**。没有单独的图像塔，没有单独的文本塔，没有 cross-attention 模块——只有 self-attention 自己去理解跨模态依赖。效果不是最优，但观念上定义了「一个流」的最简版本，以及型式上的下限。

🔗 [https://arxiv.org/abs/2102.03334](https://arxiv.org/abs/2102.03334)

## 2 · ALBEF —— *Align before Fuse: Vision and Language Representation Learning with Momentum Distillation* (Li 等, Salesforce, NeurIPS 2021)

**共形贡献**：提出「对齐在前、融合在后」的三段架构：图像编码器 + 文本编码器先各自独立过一遍（做 ITC 对比），再把两边的特征送进一个**多模态编码器**、用 cross-attention 融合（做 ITM 匹配 + MLM 填空）。三损失组合 **ITC + ITM + MLM** 第一次完整被设计出来。Momentum Distillation（EMA teacher 产生软标签）直接回应 alt-text 噪声问题。ALBEF 是 CLIP 路线和早期 VL Transformer（ViLBERT / LXMERT / UNITER）路线的**合流点**——不是抛弃对比，而是把对比作为融合的预步骤。

🔗 [https://arxiv.org/abs/2107.07651](https://arxiv.org/abs/2107.07651)

## 3 · BLIP —— *BLIP: Bootstrapping Language-Image Pre-training for Unified Vision-Language Understanding and Generation* (Li 等, Salesforce, ICML 2022)

**共形贡献**：提出 MED（Multimodal Encoder-Decoder）架构——同一个视觉 backbone 配三种文本模式（单模态编码器 / image-grounded 编码器 / image-grounded 解码器），参数**大部分共享**，只在 self-attention / cross-attention / causal mask 上切换。**理解与生成第一次被压到同一个骨干里**。同时提出 CapFilt：用模型自己生成 caption，再用模型自己过滤噪声对——**自举式数据清洗**，对 ALIGN「规模吃噪声」的另一种回应。

🔗 [https://arxiv.org/abs/2201.12086](https://arxiv.org/abs/2201.12086)

## 4 · CoCa —— *CoCa: Contrastive Captioners are Image-Text Foundation Models* (Yu 等, Google, TMLR 2022)

**共形贡献**：把 BLIP 的三模式简化为 **一个模型 + 两个损失**——图像 encoder + 文本 decoder，decoder 的**前半段做对比**（取 [CLS] 跟图像全局向量做 ITC），**后半段做 captioning**（通过 cross-attention 看图像、自回归预测文本）。一个模型同时在 retrieval / classification / VQA / captioning 四类 benchmark 上拿 SOTA——证明**对比与生成不是两条竞争路线，是互补的两个监督信号**。

🔗 [https://arxiv.org/abs/2205.01917](https://arxiv.org/abs/2205.01917)

## 5 · BEiT-3 —— *Image as a Foreign Language: BEiT Pretraining for All Vision and Vision-Language Tasks* (Wang 等, Microsoft, arXiv 2022)

**共形贡献**：提出 **Multiway Transformer**——每层有一个共享的 self-attention + 三组 modality-specific FFN expert（vision expert / language expert / vision-language expert）。训练时不同任务走不同 expert；推理时也可以按需选 expert。论点：既然 Transformer 能处理任意 token 序列，把图像 patch 当作「另一种语言」 token 喂进去就好（「image as foreign language」）。**把 §22 的「融合」推到极端**：不再有「图文两个塔」或「三个模式」的概念，只有「一个 Transformer + 按模态切换 FFN」。这条路暗接上 §26（视觉作为语言原语 / Fuyu 式）。

🔗 [https://arxiv.org/abs/2208.10442](https://arxiv.org/abs/2208.10442)

# 关于 §22 五篇的机制追问

## 一、「融合」到底是什么：让一个塔在前向时「听」另一个塔

§21 的双塔里，图像塔和文本塔在前向过程中从未相遇——只在损失处碰一次点积。§22 做的事情就是把这条隔离打破：**让一个模态的 token 在编码过程中读到另一个模态的状态**。

工程上有三种典型写法：

| 融合方式 | 机制 | 代表模型 |
| --- | --- | --- |
| **concat + self-attention** | 图文 token 拼成同一序列，交给 self-attention 自己学跨模态依赖 | ViLT / BEiT-3 |
| **双编码器 + cross-attention 融合头** | 图文先各自编码，末端加一个融合模块：文本 token 做 query，图像 token 做 key/value | ALBEF / BLIP |
| **共享骨干 + 多路 experts** | 同一条 Transformer，按模态切换 FFN | BEiT-3 |

这三种写法所回答的本质问题是同一个：**两个模态在哪一层开始相遇？cross-attention 还是 self-attention？每个模态各自多深的单模态专用层？**这个「相遇位置」的选择是 §22 的中心设计变量，也是区分五篇的主轴。

### 为什么这一步重要

回到 §21 的残差：CLIP 学到的是「相似度排序」，**两个向量之间只有一个标量关系（cosine）**。这种关系无法表达：

- 结构：它不能回答「图里哪个物体对应文本里的哪个词」。
- 关系：它不能回答「两个物体之间的空间关系是什么」。
- 组合：它不能回答「图和文在哪个细节上不匹配」。

而 cross-attention / self-attention 之所以能回答这些，是因为它们输出的不再是一个全局向量，而是**一组 token 的联合表征**——每一个文本 token 都能指向它“所关注的」那块图像区域。这也是为什么 VQA / grounding / 细粒度 retrieval 这些任务在 §22 时代精度直接跳一大截。

## 二、ALBEF 的三损失组合：为什么「对齐在前、融合在后」

### 三损失各自做什么

- **ITC（Image-Text Contrastive）**：与 CLIP 相同，在两个单塔输出上做双向对比。
- **ITM（Image-Text Matching）**：融合模块的输出里取 [CLS]，接一个二分类头——判配对 / 不配对。关键是**硬负样本挖掘**：不配对的负样本从 ITC 已经拉近的那些「似像配对的」中选，而不是 batch 里随便抽。
- **MLM（Masked Language Modeling）**：把文本 token 随机 mask，在**图像存在的条件下**从融合表征中预测被 mask 的词。这实际上是**语言模型的预训练第一次被挂在视觉条件下**——但注意，它是 BERT 式的 MLM，不是 GPT 式的自回归 LM。

### 为什么三损失必须同时挂着

每一损失只能触到任务的一层：

- 只有 ITC：就退化成 CLIP，拿不到细粒度。
- 只有 ITM：边界清楚，但需要有预对齐的特征才能找到好的硬负（否则随机抽的负太容易）。
- 只有 MLM：学到的是「看着图填词」，但没有处理非配对场景的能力。

三损失互相支持：ITC 为 ITM 提供硬负挖掘的基础； ITM 为融合模块提供「是否配对」的全局信号；MLM 为融合模块提供「结构化描述」的局部信号。**三损失加起来才把融合编码器训得成**——只加任意两个都有块残缺。这套§ITC + ITM + (M)LM§ 的组合被后面几乎所有 VL 模型继承（只是MLM 在 BLIP/CoCa 里换成了自回归 LM）。

### Momentum Distillation：用自己的历史平均去抗 alt-text 噪声

alt-text 常有「一图多配」问题：网页里同一张图对多段都算合理描述，但对比损失把非配对的那些**硬当负**。ALBEF 的做法：维护一个模型的 EMA 版本作为 teacher；让 student 除了在硬标签上训练，还在 teacher 给出的**软标签**（整个 batch 的概率分布）上训练。效果：对一张图，teacher 会给几个「也算合理」的候选都分配一些概率，student 不被硬逐到单点。

从共形视角看：ITC 的「配对的 cosine 大于所有非配对」是一个硬排序目标；MD 把它柔化为「配对的 cosine 应符合 teacher 的软分布」——这是对「共形的第一要义是保相关性」的一次很精细的工程化：相关性不一定是二值配对 / 不配对，可以是一个软的分布。

## 三、BLIP 的 MED 架构与 CapFilt：理解与生成第一次同骨干

### MED：一个视觉 backbone 配三种文本模式

BLIP 的文本部分有三个形态，三者**参数大部分共享**，只在层内配置上切换：

| 模式 | self-attention | cross-attention | mask | 用途 |
| --- | --- | --- | --- | --- |
| 单模态文本 encoder | 双向 | 无 | 无 | ITC 对比 |
| image-grounded encoder | 双向 | 开 | 无 | ITM 匹配 |
| image-grounded decoder | 因果（causal） | 开 | causal | LM captioning |

这套设计的精妙在于：**理解（编码器）和生成（解码器）被压进同一套参数**。切换任务只需要切换 mask 和 cross-attention 的开关，不需要两套独立的模型。这是阶段 III 第一次正式把「理解」和「生成」放到一个架构里。

### 但注意：BLIP 的「生成」仍严格在理解侧

这里要画一条关键分界线。BLIP 的生成是**看图说话**——它是文本生成，图像是条件。它**不能生成像素**。按阶段 III 总纲的分界：

- 能对 x 说出 y（BLIP 的 captioning）→ 属于理解侧，留 III。
- 能从零合成 x（GPT-4o 画图 / Chameleon）→ 属于生成侧，归 IV。

BLIP 的 MED 把「理解侧的生成能力」（即 captioning）收回了同一骨干——**这仍是 §22 的范围**，跟 IV 的「生成像素」没有关系。不要把 BLIP 误读为「阶段 III 已经包含生成共形」。

### CapFilt：用模型自己洗自己的数据

做法：

1. 用初步训练好的 decoder 对每张图**生成**一个新 caption——比 alt-text 丰富。
2. 用 ITM 对每一对（原 alt-text + 生成 caption）做配对性**打分**，剩下高分的。
3. 用这个清洗后的数据集重新训练模型。

它是对 ALIGN「规模吃噪声」的另一种回应：不是靠数据量稀释噪声，是靠**模型自己改写噪声**。这在很多层面上都算阶段 III 内部的小型反身芽：模型对自己的输入做质量给出评判。但这还远不是反身度量（留给 V）——它只是「模型对数据质量做分类」，不是「模型质疑自己的度量」。

## 四、CoCa 的双损失统一：为什么对比与生成是互补的

### 架构

- 图像 encoder（ViT）→ 全局 token
- 文本 decoder（causal self-attention + cross-attention），分为前后两段：
    - **前半段**（只有 causal self-attention，没有 cross-attention）：当作单模态文本 encoder，拿最后的 [CLS] 和图像全局向量做对比（ITC）。
    - **后半段**（causal self-attention + cross-attention）：看图生成 caption（LM）。

一个模型，两个损失，共享前半参数。

### 为什么只有 contrastive 不够、只有 captioning 也不够

两个损失在**语义粨度**上是互补的：

|  | 对比 (ITC) | 生成 (LM) |
| --- | --- | --- |
| 粨度 | 粗（单个全局向量） | 细（逐 token 预测） |
| 监督信号形状 | 排序 / 区分 | 生成 / 复构 |
| 擅长任务 | retrieval / classification | VQA / captioning |
| 相对于 alt-text 噪声 | 敏感（需 MD 或 CapFilt） | 不敏感（逐 token 主频主导信号） |
- 单做 ITC：学到的是「识别」，拿不到「描述」。
- 单做 LM：学到的是「描述」，但 retrieval 的结构化空间没有对齐（出去的 [CLS] 不是为 retrieval 训的）。
- 两者并提：既有可用于 retrieval 的对齐向量，也有与图像对齐的描述能力。

这一点对后续影响极大：§23 之后的 LLM-集成式 VLM（BLIP-2 / LLaVA 一系）训练时通常同时挂对比损失和自回归 LM——带的就是 CoCa 这个「互补而非竞争」的结论。

### 从共形视角重写这一并提

- ITC 是「保相关性」（相似度排序对齐）。
- LM 是「保生成链」（在条件下逐 token 重建文本）。
- 两者对应共形原理的两个不同投影——一个负责整体结构在两个空间之间的协变形，一个负责细节在给定条件下被重新生成。**两者并用相当于在粗粒度与细粒度两层同时执行共形压力**。

## 五、BEiT-3：统一骨干的终点 ——也是 §22 极限的暴露

### Multiway Transformer

每一层：

- 一个共享的 self-attention（所有 token 走同一套 QKV）。
- 三组 modality-specific FFN experts：
    - V-expert：处理图像 token。
    - L-expert：处理文本 token。
    - VL-expert：处理跨模态 token（用于在 top layer 做融合）。

训练：单图任务（MIM / image classification）走 V-expert；单文任务（MLM）走 L-expert；图文对任务下层走各自 expert、上层走 VL-expert。

### 为什么这叫「融合的极限」以及「极限的暴露」

- 极限：没有专用的 VL 融合模块，只有「一条 Transformer + 按模态切换 FFN」。从架构上讲，不能再更简了。
- 暴露：这件事同时说明，**基底 Transformer 在 2023 左右已经不重要**——重要的是你多大、你用多少数据、你训了多久。BEiT-3 的骨干规模是 1.9B；同期的 LLaMA 是 7–70B，GPT-3 是 175B。只要 §22 的路线还守在「从头训练一个 VL 专用骨干」上，它永远追不上 LLM 侧的缩放红利。

这就是 §22 的路线性极限。**融合做的再好，只要是「从头训 VL Transformer」，就不可能赶上 LLM 的步伐**。§23 的切入点就在这里：不再从头训 VL 骨干，而是直接把 LLM 冻住、用一个小模块把视觉接进去。BEiT-3 在架构上的优雅恰好是 §22 路线困形的证明。

## 小结：§22 的谱系内逻辑

- **ViLT**：把「一个流」的最单纯形写出来——**concat + self-attention**。观念上的下限，但精度不是最优。
- **ALBEF**：提出 **ITC + ITM + MLM 三损失**，通过 Momentum Distillation 处理噪声——「对齐在前、融合在后」的标准范式定型。
- **BLIP**：提出 **MED 架构**，把理解与生成压进同一条骨干；CapFilt 做自举式数据清洗。
- **CoCa**：把三模式简化成 **对比 + captioning 双损失**，一个模型覆盖四类 benchmark。
- **BEiT-3**：把融合推到极限——**共享 Transformer + 多路 experts**；但同时暴露 §22 路线永远追不上 LLM 规模的结构性困境。

五篇合起来完成一件事：**让两个塔在编码器层面相遇，并把这种相遇的工程形式从最简（ViLT）推到最精（ALBEF）再推到最统一（BEiT-3）**。但也同时暴露：这条路上，骨干又被锁在了「专用 VL Transformer」里。

---

## 本组合力的残差

融合在编码器层面做实了，但暴露三条新残差：

1. **专用 VL Transformer 吃不下 LLM 缩放红利**——ALBEF / BLIP / CoCa / BEiT-3 的文本部分都是从头训练或从 BERT 级预训练起步，不是 GPT-3 / PaLM / LLaMA 级别。当 LLM 走到 100B+ 规模时，这些专用模型的推理能力无法跟上。这条残差**定义 §23 的入口**：不再从头训 VL 骨干，而是**冻住现有 LLM，用一个小适配层把视觉接进去**（Flamingo / BLIP-2 / Q-Former）。
2. **生成仍是 caption 级，不是指令级**——BLIP 和 CoCa 能生成，但能生成的只是「看图说话」；它们不能被「任意指令」提示（「比较两张图的差异」、「找出这张表格的错误」这类指令无法响应）；也没有 in-context learning。这条残差**定义 §24 的入口**：视觉指令微调（LLaVA / MiniGPT-4 / InstructBLIP）。
3. **融合是 VL 任务专用，不是通用推理平台**——这些模型擅长 VQA / captioning / retrieval / grounding 这些 VL 专用 benchmark，但无法像 LLM 一样被任意 prompt。例如你不能让 BLIP「写一段关于这张图的诗」、也不能让 CoCa「按这张图的风格写故事」。**这两种能力对应的不只是监督数据缺失，而是底层 decoder 不是一个被花样指令训练过的 LM**。这条残差也交给 §23–§24，它是第 1 条和第 2 条的混合产物。

第 1 + 3 条定义 §23 的入口；第 2 条定义 §24 的入口。**§22 是「从两个塔到一个流」的完成，也是「但这个流还不够大」的开始**。§23 之后的路子不再是「把 VL 骨干做更精」，而是「放弃自己训练 VL 骨干，跟 LLM 这条主线对接」。
