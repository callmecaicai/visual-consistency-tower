# §24 · 机制展开：统一多模态骨干的收敛与代价

本页补足 §24 的机制承重。统一多模态骨干的意义不是“一个模型打多个 benchmark”，而是为视觉接入语言后端准备标准接口。

---

## 一、三类统一

| 类型 | 代表 | 统一了什么 | 代价 |
|---|---|---|---|
| 多目标 VLP 骨干 | CoCa、BLIP | 对比、匹配、caption 等训练目标 | loss 配比与数据配方复杂 |
| Seq2Seq 任务统一 | OFA、Unified-IO、Florence-2 | 把任务输出序列化为 token | 接口统一但状态不统一 |
| 冻结 LLM 桥接 | Flamingo、BLIP-2、LLaVA | 把视觉 token 接到语言推理后端 | 视觉带宽受 connector 与上下文限制 |

§24 的核心不是宣布 Seq2Seq 胜出，而是说明：一旦任务都能被写成语言接口，视觉就可以进一步退化为供 LLM 调用的前端。

## 二、BLIP-2 的桥接地位

BLIP-2 的 Q-Former 用一组可学习 query 从冻结视觉编码器中抽取固定数量视觉 token，再喂给冻结 LLM。它不是“首次”冻结语言模型，但它把桥接模块的功能写得非常清楚：

1. 视觉编码器可以复用强预训练模型。
2. LLM 可以保持冻结，避免昂贵的多模态全量预训练。
3. query 数量成为视觉带宽的显式控制旋钮。

它与 DETR 的 object query 有结构回声：query 不再只是输入 token，而是从 dense representation 中抽取任务相关信息的接口。

## 三、Florence-2 的 prompt-as-task

Florence-2 把 caption、OD、OCR、grounding、segmentation 等任务用 task prompt 触发，并用结构化 token 输出结果。它是 prompt-as-task 的极端代表：任务头被弱化，接口被语言化。

但这种统一仍是接口统一，不是世界状态统一。系统能用统一 token 格式交付多个任务结果，不等于内部持有对象、几何、遮挡和时序状态。

## 四、出口

§24 的残差自然推向 §25：

1. 统一骨干的 text decoder 仍然浅，语义推理需要 LLM 后端。
2. 多任务接口统一不等于视觉主体性。
3. caption / VQA / grounding 仍在理解侧，不能反向生产观测。
