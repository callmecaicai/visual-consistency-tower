# §22 · 机制展开：CLIP 为什么是范式置换而不是改良

本页补足 §22 的机制承重。CLIP 的核心不是“多了一个文本塔”，而是把互联网上的人类描述当作视觉监督材料，并让图像与文本共享一个可比较的公共相似度空间。

---

## 一、监督经济学置换

外挂范式依赖 bbox、region tag 与 caption 的精标组合，数据质量高但扩张慢。CLIP 把监督单位改成：

```text
image-text pair
```

它牺牲 region 级精度，换取 web-scale 数据半径。这是监督经济学置换：不是更精的监督，而是更大的公共语义材料。

## 二、InfoNCE 的三层读法

| 层级 | 含义 |
|---|---|
| 技术层 | batch 内形成 N x N 图文相似度矩阵，对称地做 image->text 与 text->image 匹配 |
| 机制层 | 正配对被拉近，负配对被推远，图像和文本被迫进入可比较空间 |
| 本体层 | 视觉差异第一次被大规模公共语言坐标系约束 |

InfoNCE 可以和互信息下界联系起来，但 CLIP 的工程成效不能简化成“最大化互信息”。温度参数更稳妥地说，是相似度分布 sharpness / concentration 的控制器；除非给出明确几何定义，不应直接称为“曲率控制”。

## 三、三个工程赌注

| 赌注 | 为什么成立 | 代价 |
|---|---|---|
| 全图级对齐够用 | caption 与图像整体语义在 web-scale 下可形成强公共语义场 | 空间、计数、方向、遮挡被压扁 |
| 噪声标签可容忍 | 大 batch 对比目标对单个错配有一定平均化能力，规模能抵消部分噪声 | 噪声分布会变成语义偏见 |
| 双塔解耦 | 文本侧可缓存，图像侧可独立检索和复用 | 两塔只在相似度空间相遇，不产生共享世界状态 |

## 四、家族定位

| 工作 | 推进了什么 | 留下什么 |
|---|---|---|
| CLIP | 建立 web-scale 双塔对比范式 | 全图向量丢空间 |
| ALIGN | 扩大噪声数据规模 | 数据噪声更重 |
| LiT | 冻结强视觉塔，复用视觉表征 | 对齐依赖已有视觉基础 |
| OpenCLIP / LAION | 开源化数据与训练 | 复现数据质量参差 |
| EVA-CLIP | 用 MIM 强化视觉塔 | 仍是语言友好空间 |
| SigLIP | 用 sigmoid loss 缓解 softmax 温度和 batch 问题 | 不解决 caption 盲区 |
| MetaCLIP | 重建数据 query balancing | 说明数据配方是核心变量 |

## 五、为什么 CLIP 不是稠密闭合

CLIP 的对齐单位是全图与整句。它天然扩大公共语义半径，却不保证 pixel / region / instance 的局部闭合。方向、计数、局部关系和状态等属性，如果没有被 caption 稳定写入，就很容易在全图相似度空间中被压扁。

这正是 §23 出现的原因：语言必须从全图相似度进入稠密输出链。

## 六、账单种子

CLIP-blindness 不是 §27 才突然出现的故障，而是在 §22 的成功中埋下的账单。它来自三重压缩：

1. caption 只描述人类觉得值得说的内容。
2. 全图单向量压缩掉局部结构。
3. 语言友好空间优先保留可命名差异，而不是全部视觉差异。

因此，CLIP 同时是阶段 III 的入场券和债主：它让视觉进入公共语义，也规定了哪些视觉差异会被公共语义忽略。
