# 08-命名制度与文件职责协议

一句话定位: 文件名负责定位，标题负责思想，副标题负责命运。

关键贡献: 给整座视觉理论大厦建立统一命名制度，区分正式主线、横切协议、前沿补遗、施工记录与归档材料，避免文件名继续承担过多叙事压力。

定位标签: 命名制度, 文件职责, 导航显示, 施工记录, 主线治理

---

## 一、根原则

命名分两层。

文件路径名必须功能性、稳定、短、可检索。

页面标题可以浪漫、精确、有思想密度。

因此，一个页面应当形成三层结构：

| 层级 | 职责 | 示例 |
|---|---|---|
| 文件名 | 稳定定位 | `§34-轨迹闭合-扩散与内部时间.md` |
| 页面标题 | 表达命题 | `§34 轨迹闭合：样本如何在内部时间中显形` |
| 副标题 | 说明命运 | `阶段 IV 的关键不是会生成，而是生成过程第一次获得可审计轨迹。` |

禁止把完整论文标题、长哲学句、导入 hash、临时计划名塞进文件名。

## 二、文件角色类型

所有 Markdown 文件先归入角色，再命名。

| 角色 | 功能 | 推荐命名 |
|---|---|---|
| `root_manifest` | 首页、总入口 | `INDEX.md` |
| `axiom` | 公理、开端、根命题 | `第一性原理.md` |
| `core_grammar` | 阶段核心语法 | `01-核心语法-状态等变预测反身.md` |
| `stage_overview` | 阶段总纲 | `00-总纲-视觉作为投影表面存在.md` |
| `position_card` | 主章定位卡 | `00-定位卡.md` |
| `mechanism` | 技术机制展开 | `01-机制展开.md` |
| `deepening` | 追问、深化、哲学映射 | `02-追问与深化.md` |
| `evidence_book` | 证据、账本、失败案例 | `02-度量账本-生成评价与指标残差.md` |
| `frontier_update` | 前沿补遗 | `03-前沿补遗-2026.md` |
| `closing` | 阶段结语 | `结语-空间胜利与意义空洞.md` |
| `appendix` | 附录、补章 | `附录A-材料制度与训练工艺.md` |
| `navigation` | 导航入口 | `90-阶段导航.md` |
| `study_map` | 学习地图 | `91-学习地图-阶段IV怎么学.md` |
| `debt_record` | 债务与待补 | `92-阶段IV债务与待补.md` |
| `construction_log` | 施工计划、优化记录 | `90-施工记录/2026-阶段III优化记录.md` |
| `archive` | 历史旧稿 | `91-历史审计/旧版总审计.md` |
| `domain_instance` | 应用域实例 | `00-总纲-时间差异视觉的第一实例.md` |
| `tooling` | 工具说明 | `README.md` |

无法归类的文件，先不进入正式导航。

## 三、顶层导航显示名

真实目录名保持短、稳定，导航显示名承担诗性和精确度。

| 真实目录 | 导航显示名 |
|---|---|
| `00-公理层` | 开端与公理：可见性如何生成视觉世界 |
| `01-阶段I-判别共形` | 阶段 I：视觉作为标签存在 |
| `02-阶段II-稠密共形` | 阶段 II：视觉作为投影表面存在 |
| `03-阶段III-语言共形` | 阶段 III：视觉作为公共语义契约存在 |
| `04-阶段IV-生成共形` | 阶段 IV：视觉作为外显生产存在 |
| `05-阶段V-表征共形` | 阶段 V：视觉作为部分世界状态存在 |
| `横切` | 横切协议：残差、反身与预测登记 |

`RS_CD` 已移出本仓库的正式版本跟踪，不再作为本大厦 Git 主线发布对象。

## 四、阶段主章命名

主章文件名使用：

```text
§编号-闭合功能-技术对象.md
```

推荐模式：

| 阶段 | 主章命名主语 | 示例 |
|---|---|---|
| 阶段 I | 标签裁决、训练稳定、差分动力 | `§05-差分动力-状态保留与残差更新.md` |
| 阶段 II | 投影表面、场闭合、对象闭合、接口临界 | `§14-绑定制度-Anchor-free与Query.md` |
| 阶段 III | 命名、指代、调用、遮蔽、语言后端 | `§25-语言后端-VLM与视觉证据带宽.md` |
| 阶段 IV | 可能性、可信性、内部语、轨迹、生产制度 | `§34-轨迹闭合-扩散与内部时间.md` |
| 阶段 V | 样本同一、基础特征、预测状态、反身审计 | `§47-反身审计-度量危机与终极残差.md` |

模型名可以出现，但不能成为唯一主语。文件名应优先说明该模型在闭合制度中承担什么功能。

## 五、横切层命名

横切层只放正式协议、登记簿和审计规则。

正式横切文件应承担稳定职责：

| 文件 | 职责 |
|---|---|
| `反身脊柱.md` | 五阶段度量反身主线 |
| `残差生命周期与登记协议.md` | 残差从候选到吸收的制度 |
| `阶段偏序与桥接节点.md` | 非线性阶段关系 |
| `中层形式语法与深化协议.md` | 公理到任务的中层语法 |
| `知识库编号与索引协议.md` | 编号与索引制度 |
| `08-命名制度与文件职责协议.md` | 本文件 |
| `预测性判据登记簿.md` | 理论预测登记 |
| `预测验证协议.md` | 预测如何被证伪 |
| `领域实例化协议.md` | 应用域如何进入大厦 |

施工计划、阶段优化记录、历史审计，不进入横切正式导航。

## 六、禁止命名清单

以下命名不得出现在正式导航中：

| 类型 | 示例 | 处理方式 |
|---|---|---|
| 临时计划名 | `2026-阶段I施工记录.md` | 放入施工记录或隐藏 |
| 空泛版本名 | `最终版.md`、`最新.md` | 改为功能名 |
| 导入残留 | `临时导入包`、`Untitled` | 归档或删除 |
| 旧结构名 | `05-两条DINO线.md` | 拆成准确谱系 |
| 应用域圈号编号 | `⑤-A`、`⑥-B`、`⑦-A` | 改为应用域子层 |
| UUID/hash 文件名 | `...d28bfe9.md` | hash 放入正文元数据 |
| 过长模型串 | `Florence2-DepthAnything-RADIO-SAMencoder...` | 文件名缩短，标题展开 |

## 七、新文件创建检查表

新建文件前必须回答：

1. 它属于哪个角色类型？
2. 它是否进入正式导航？
3. 它属于哪一阶段或横切协议？
4. 它承担的是总纲、核心语法、机制、证据、前沿、结语还是施工记录？
5. 文件名是否短于 80 个字符？
6. 文件名是否包含临时词、hash、旧编号或完整论文标题？
7. 页面标题是否补足了文件名没有承载的思想表达？
8. 如果移动文件，内部链接是否已同步更新？

---

## 八、执行顺序

命名治理分两轮。

第一轮只改阅读层：

1. 修改 MkDocs 导航显示名；
2. 隐藏施工记录和归档残留；
3. 新增本命名协议；
4. 不大量移动文件，避免链接震荡。

第二轮再改路径层：

1. 对 P0 污染项执行 `git mv`；
2. 更新内部链接；
3. 运行站点 strict 构建；
4. 再提交。

当前仓库先执行第一轮。第二轮应在生成全仓库命名清单之后进行。
