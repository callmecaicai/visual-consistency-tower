# 知识库编号与索引协议

本协议不是写作修辞，而是给后续轻量查看器、索引脚本与 agent 消费的结构接口。

---

## 一、目标

1. 人可以顺着目录直接读出“这篇文档处于哪一层、哪一章、扮演什么角色”。
2. 机器只看路径与文件名前缀，就能初步判断阶段、章节、文档角色和优先级。
3. 新论文、新项目需求进入体系时，可以先被快速定位，再决定写入哪一层。
4. 阶段 IV、V 在展开前就预留编号带宽，避免后期重排。

## 二、顶层目录角色

- `00-公理层`：公理、术语、阶段总览，属于全楼地基。
- `01-阶段I-判别共形` 到 `05-阶段V-表征共形`：五个主楼层。
- `横切`：贯穿全楼的协议、审计、反身、预测。
- `应用域`：论文卡、案例库、领域专题；不直接改写主楼层，但要能反哺主楼层。
- `skills`：agent 的主运行时；负责新论文定位与知识库增量更新。
- `工具`：查看器、脚本、索引器、自动化接口；不纳入理论主线。
- 各类临时导入包、临时合并备份：不作为正式入口；若不再承担源文档角色，应从主仓库移除或保持忽略。

补充判断：

- `skills` 是主接口。
- `工具` 是辅助接口。
- 若二者冲突，优先保证 `skills` 的流程正确，再考虑工具是否需要跟进。

## 三、Agent 分层读取协议

这个仓库不是给 agent 一次性通读的，而是给 agent 分层检索的。

标准读取顺序应当固定为：

1. `T0`：全局入口层  
   `INDEX.md`、`00-总纲`、`00-阶段导读`、`90-*` 协议/导航
2. `T1`：定位层  
   `横切` 协议文档、各章 `00-定位卡`
3. `T2`：主机制层  
   `01-机制展开`、`01-总纲扩展`、阶段结语
4. `T3`：扩展证据层  
   `02+` 扩展稿 / 专题 / 补章
5. `T4`：档案与案例层  
   `应用域`、`归档`

原则：

- agent 不应默认读取整个阶段目录。
- 只有在上层无法判定时，才下钻到下一层。
- 新论文 / 新项目先落在 `T0/T1` 完成阶段与章节定位，再决定是否需要 `T2/T3`。
- `T4` 只能作为证据补充，不能替代主线定位。

## 四、阶段编号带宽

为保证阶段间可扩展，主章节统一按十位段预留：

| 阶段 | 章节带宽 | 当前状态 |
|---|---|---|
| I | `§01-§09` | 已使用 `§01-§05` |
| II | `§11-§19` | 已使用 `§11-§16` |
| III | `§21-§29` | 已使用 `§21-§27` |
| IV | `§31-§39` | 已在总纲中预告 `§31-§37` |
| V | `§41-§49` | 现阶段预留，后续展开时启用 |

这套十位分段是合理的，应当保留，不建议改成连续无分段编号。

## 五、阶段目录内部协议

阶段目录内，从现在起统一采用下面的角色前缀：

- `00-*`：阶段正式入口。
  例：`00-总纲-*`、`00-阶段导读.md`
- `01-*`：阶段级正式扩展。
  例：`01-总纲扩展-*`
- `§xx-*`：主章节目录或单文件章节。
- `补章*`：阶段补充专题，只在确有必要时使用。
- `结语*`：阶段收束。
- `90-*`：导航、归档、结构性辅助文件，不参与正文推进。

补章内部也要可索引：

- `00-总览.md`
- `A1/B1/...` 或 `01/02/...` 二选一，但同一补章内部不得混用。
- 不再新增无前缀的补章内部文件。

## 六、章节目录内部协议

从阶段 IV 起，新章节默认使用目录形态，不再优先写成单文件。

标准章节目录：

1. `00-定位卡.md`
2. `01-机制展开.md`
3. `02-扩展稿-*.md`
4. `03-专题-*.md`
5. `90-附录-*.md` 或 `90-索引-*.md`，仅在必要时使用

语义规则：

- `00`：这章在全楼里解决什么残差、承接谁、把谁送出去。
- `01`：本章正式主机制，是引用时的第一正文。
- `02+`：证据、长稿、专题、案例、补遗，只能服务于 `00/01`，不能反客为主。
- `90+`：不直接参与理论论证的辅助材料。

## 七、最小元数据头

为了让查看器和 agent 不靠全文猜测，新的正式文档建议在前 20 行内具备最小头信息：

```md
# 标题

类型: 理论/分析/论文卡/专题/协议
阶段: III · 语言共形
共形维度: 载体, 主体性
关键贡献: ...
残差 / 它催生了什么: ...
定位标签: CLIP, 对比对齐, VLM
```

允许保留现有写法，但从现在起新增正文最好补上 `定位标签` 这一行，方便索引器和定位器直接消费。

## 八、新论文 / 新需求的落位协议

当一篇新论文或一个新项目需求来到面前，先不要直接问“它像哪篇 paper”，而要问下面六件事：

1. 它的输入输出是什么。
2. 它主要解决的是判别、稠密、语言、生成，还是表征预测问题。
3. 它改动的主轴是材料、动力、载体、度量、时间尺度，还是主体性。
4. 它是在回应上一阶段的残差，还是在本阶段内部推进。
5. 它更像主机制，还是应用侧证据。
6. 它会把什么新残差送给下一章、下一阶段。

若要写成可被 agent 快速消费的卡片，最少应补四项：

- `阶段候选`
- `章节候选`
- `共形维度`
- `上游残差 / 下游出口`

## 九、当前 I-III 的一致性判断

已经合理的部分：

- 主阶段采用 `01 / 11 / 21` 十位分段，扩展性足够。
- 阶段 III 已基本进入可机读状态：`00-总纲`、`01-总纲扩展`、`§21-§27/00/01/02+`。
- `横切` 与 `应用域` 已经和主楼层分离，这是对的。

仍属遗留、但暂不强拆的部分：

- 阶段 I 的 `§03`、`§04` 仍是单文件；阶段 II 的 `§11`、`§13` 也是单文件。
- 阶段 I 的 `补章A`、`补章B` 内部编号不连续，且仍有无前缀文件。
- 少量旧文档只有标题，没有稳定的元数据头。

当前决策：

- I-III 先保留遗留写法，不做大规模重命名。
- 阶段 IV、V 从开始起严格采用目录化章节协议。
- 若未来回头清洗 I-III，优先处理补章与单文件章节，而不是动主章节带宽。

## 十、对轻量查看器的直接要求

查看器最少要能消费下面这些结构信号：

- 顶层目录角色
- 阶段号与章节号
- 文档角色前缀（`00/01/02+/90`）
- 分层读取层级（`T0-T4`）
- 是否归档、是否应用域、是否横切
- 最小元数据头

只要这些信号稳定，后续无论是静态浏览器、检索脚本，还是 agent 自动定位器，都可以共用同一套底层索引。

## 十一、对 Skills 的直接要求

`skills/` 应视为这座知识库的第一运行时，而不是附属文档。

至少应常驻两类 skill：

1. `paper-positioning`
   负责“新对象先落位”
2. `kb-incremental-update`
   负责“落位之后再更新”

固定顺序：

1. 先定位
2. 再决定读取深度
3. 再决定写入位置
4. 最后才动 Markdown
